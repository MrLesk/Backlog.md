## Findings

### important — watcher-lock fallback still double-fires across processes
- **Location:** [src/core/backlog.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/backlog.ts:213), [src/core/backlog.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/backlog.ts:1169), [src/server/index.ts](/D:/1064n/Programacion/claude/Backlog.md/src/server/index.ts:300), [src/core/task-hook-dispatcher.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/task-hook-dispatcher.ts:139), [src/cli.ts](/D:/1064n/Programacion/claude/Backlog.md/src/cli.ts:3912)
- **Issue:** The suppression mechanism is process-local, but `dispatchInProcess()` still runs in every `Core` instance while only one process owns the watcher lock. That means process B can perform an in-process status update and fire the hook immediately, then process A (the lock holder with the active watcher) observes the same file change later and fires again because its own coordinator never saw B's `beginWrite`/`endWrite`. The browser-server lock fallback comment says “the lock holder drives hook dispatch for everyone”, but the actual code only disables watchers on the non-holder; it does not disable in-process dispatch there. The same duplicate can happen with `backlog watch` in one process and any CLI/MCP/browser write in another.
- **Suggested fix:** Pick one authority per project for hook dispatch and enforce it. The simplest fix is: if a process does not hold the watcher lock, it must not call `dispatchInProcess()` for task writes. That likely means making lock ownership visible on `Core` and gating all in-process dispatch sites on it, or moving hook dispatch authority entirely to the lock holder via an IPC/queue mechanism instead of relying on per-process suppression state.

### important — 2-second suppression window makes the “no double-fire” guarantee probabilistic
- **Location:** [src/core/task-write-coordinator.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/task-write-coordinator.ts:53), [src/core/task-write-coordinator.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/task-write-coordinator.ts:106), [src/core/content-store.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/content-store.ts:649), [src/core/content-store.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/content-store.ts:908)
- **Issue:** The coordinator suppresses any same-task watcher event for a fixed 2-second period after `endWrite()`. That creates both failure modes the design was supposed to avoid: a legitimate hand edit within 2 seconds of an in-process write is incorrectly suppressed, and a delayed `fs.watch` event or slow read/retry path can arrive after 2 seconds and get treated as a hand edit, re-firing the hook for the original in-process write. `retryRead()` alone can stretch well past 2 seconds on exhausted retries, so this is not just theoretical. As implemented, the guarantee is “usually no double-fire on this machine”, not “no double-fire”.
- **Suggested fix:** Replace the time-window heuristic with a write identity the watcher can match deterministically. For example: record file `mtime`/size/content hash captured at write time and suppress only the watcher event that resolves to that exact on-disk version, or keep the write handle live until the corresponding watcher observation is consumed instead of expiring it on a timer.

### important — reinitializing the project root leaves the dispatcher bound to the old root and stale snapshot state
- **Location:** [src/core/backlog.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/backlog.ts:213), [src/core/backlog.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/backlog.ts:556), [src/core/task-hook-dispatcher.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/task-hook-dispatcher.ts:80), [src/core/task-hook-dispatcher.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/task-hook-dispatcher.ts:115), [src/mcp/server.ts](/D:/1064n/Programacion/claude/Backlog.md/src/mcp/server.ts:229), [src/mcp/server.ts](/D:/1064n/Programacion/claude/Backlog.md/src/mcp/server.ts:266)
- **Issue:** `Core` constructs `taskHookDispatcher` once with `cwd: this.fs.rootDir` and a private `previousStatus` map, then `reinitializeProjectRoot()` swaps `this.fs`/`this.git` without rebuilding or resetting the dispatcher/coordinator. After an MCP roots upgrade/downgrade, hook subprocesses still run with the old `cwd`, and the snapshot map still contains task statuses from the previous project/store lifetime. `seedSnapshot()` only overwrites seen tasks; it does not clear removed ones. That can produce wrong working directories for hook commands and stale status comparisons after project switching or dispose/re-init.
- **Suggested fix:** Recreate the dispatcher/coordinator inside `reinitializeProjectRoot()`, or add an explicit `reset({ cwd })` on the dispatcher that clears the snapshot and updates the execution root before any new `ContentStore` is created.
