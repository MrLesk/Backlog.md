## Findings

### important — hash-based suppression still races because writes are recorded after the file change is already observable
- **Location:** [src/core/content-store.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/content-store.ts:336), [src/core/content-store.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/content-store.ts:412), [src/core/content-store.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/content-store.ts:649), [src/file-system/operations.ts](/D:/1064n/Programacion/claude/Backlog.md/src/file-system/operations.ts:326), [src/core/backlog.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/backlog.ts:2658)
- **Issue:** The new content-hash design fixes the old 2-second heuristic in principle, but the implementation records the hash only after `originalSaveTask()`/`Bun.write()` has already completed. That leaves a real race window where `fs.watch` can deliver the event first, the watcher hashes the new bytes, `consumeMatchingWrite()` returns `false` because nothing has been recorded yet, and the watcher fires as a hand edit. The in-process path then later reaches `dispatchHookInProcess()` and fires again. This affects both the wrapped `saveTask()` path and `editTaskInTui()`. The old begin/end bracket avoided exactly this timing hole; the new code reintroduces it.
- **Suggested fix:** Record the exact bytes before the watcher can observe them. For `saveTask()`, the straightforward fix is to surface the serialized task content from `FileSystem.saveTask()` (or a shared serializer helper) so the wrapper can hash the same bytes before/while writing, without a post-write readback gap. For `editTaskInTui()`, compute the hash for `withUpdatedDate` and record it before `await Bun.write(...)`, or otherwise make the record/write step atomic from the watcher's perspective.

### important — project-root reinitialize still leaves hook-authority state bound to the old project
- **Location:** [src/core/backlog.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/backlog.ts:261), [src/core/backlog.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/backlog.ts:274), [src/core/backlog.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/backlog.ts:311), [src/core/backlog.ts](/D:/1064n/Programacion/claude/Backlog.md/src/core/backlog.ts:637), [src/mcp/server.ts](/D:/1064n/Programacion/claude/Backlog.md/src/mcp/server.ts:229), [src/mcp/server.ts](/D:/1064n/Programacion/claude/Backlog.md/src/mcp/server.ts:266)
- **Issue:** Round 8’s dispatcher/coordinator stale-state problem is only partially fixed. `reinitializeProjectRoot()` resets the dispatcher snapshot and pending write hashes, but it does not reset `hookAuthorityResolution` or release `hookAuthorityLockHolder`. If this `Core` has ever lazily probed hook authority in project A, then after an MCP roots upgrade/downgrade to project B it can keep reusing the cached authority result from A, and can also keep holding A’s watcher lock. Subsequent `dispatchHookInProcess()` calls in B may therefore skip probing B’s lock entirely and make fire/suppress decisions using stale cross-project state.
- **Suggested fix:** Treat hook-authority state as project-scoped and clear it on root swaps. `reinitializeProjectRoot()` should release any lazy-acquired authority lock and null out the cached resolution before the new project starts doing updates. If synchronous release there is awkward, add an async reinit path or a dedicated internal reset helper that safely tears down authority state before the next dispatch.
